# /要件对照表

1. 加载 `~/.claude/plugins/config/claude-for-legal/litigation-legal/CLAUDE.md` → 角色、工作成果抬头、决策姿态、文件存储。
2. 遵循以下工作流程和参考。
3. 模式选择：
   - `--专利` → 专利权利要求对照表（侵权比对表/无效比对表）。需要专利号和至少一项主张的权利要求。子模式：`--侵权`、`--无效`、`--审查`。
   - `--民事` → 民事要件事实对照表。需要诉因（或抗辩）和立场。
   - 无标志 → 询问用户选择。
4. 民事模式：参考技能目录中的 `references/民事要件模板.md` 获取基准要件列表。映射前与用户确认适用的裁判要旨或法条。
5. 专利模式：将主张的权利要求解析为技术特征，标注争议术语待解释，适用已有权利要求解释裁定。
6. 将要件映射至目标（被控侵权产品/现有技术/证据集/待审查对照表）。每格附精确引用。
7. 产出缺口清单（民事）或待证清单（专利）——优先输出。
8. 写入Markdown、CSV（含`_来源`配套文件）以及Excel或表格。
9. 如有立案案件且已启动，写入该案件文件夹；否则写入执业层面的文件夹。追加一条至 `办案日志.md`。
10. 返回摘要读数：权利要求/诉请、目标、管辖、阶段、按状态的要件数、缺口清单、文件路径。

# 要件对照表

## 举证材料使用限制

在处理一组诉讼文件前，询问："这些文件中是否有通过证据交换程序从对方当事人获取的？"如果是：

- 确认："本次使用属于本案诉讼程序范围内，或已取得对方同意/法院许可，或文件现已公开。"如果未确认，标注："证据交换文件可能存在使用限制。继续前请确认本次使用被允许。"

## 对照表是初稿，不是结论或法律文书

**将此置于每项输出顶端。不得省略。不得弱化。**

> 本对照表是供律师分析和核实的初稿，不是已提交的起诉状、代理词、证据目录或法律意见。每条映射均为律师须按来源核实的线索。所列要件来自示范性裁判要旨、民法典相关条文或经解析的权利要求语言——用户所在管辖区的**有约束力**规范（最高人民法院司法解释、指导性案例、适用的法条、已生效的权利要求解释裁定）可能有差异且始终为准。缺口检测是举证或法律文书的出发点，不是对实体问题的结论。

漏标缺口是单向门——诉状对某要件缺乏事实基础、庭审对争议要件缺少证据、案件缺乏侵权行为证明。多标缺口是双向门——律师在审查中清除标注。默认偏向双向门。

开头先问：

> 哪种对照表？
>
> 1. **专利权利要求对照表** —— 权利要求技术特征逐项对比被控侵权产品（`--侵权`）、现有技术（`--无效`）或对方对照表（`--审查`）。

- **已有对照表？** 如为 `--审查`，加载之。

# 模式1 —— 专利权利要求对照表（专利侵权比对表）

## 法律依据

- **全面覆盖原则**：《专利法》第64条——发明或者实用新型专利权的保护范围以其权利要求的内容为准，说明书及附图可以用于解释权利要求的内容。被控侵权技术方案包含与权利要求记载的全部技术特征相同或者等同的技术特征的，应当认定其落入专利权的保护范围。
- **等同原则**：《最高人民法院关于审理侵犯专利权纠纷案件应用法律若干问题的解释》第7条——被控侵权技术方案的技术特征与权利要求记载的全部技术特征相比，缺少权利要求记载的一个以上的技术特征，或者有一个以上技术特征不相同也不等同的，应当认定其没有落入专利权的保护范围。
- **等同特征的认定**：《最高人民法院关于审理专利纠纷案件适用法律问题的若干规定》第13条——等同特征是指与所记载的技术特征以基本相同的手段，实现基本相同的功能，达到基本相同的效果，并且本领域普通技术人员在被诉侵权行为发生时无需经过创造性劳动就能够联想到的特征。
- **禁止反悔原则**：《最高人民法院关于审理侵犯专利权纠纷案件应用法律若干问题的解释》第6条——专利申请人、专利权人在专利授权或者无效宣告程序中，通过对权利要求、说明书的修改或者意见陈述而放弃的技术方案，在侵犯专利权纠纷案件中又将其纳入专利权保护范围的，人民法院不予支持。
- **捐献规则**：《最高人民法院关于审理侵犯专利权纠纷案件应用法律若干问题的解释》第5条——仅在说明书或者附图中描述而在权利要求中未记载的技术方案，视为捐献给社会。

## 子模式

- `--侵权` —— 权利要求技术特征 vs. 被控侵权产品（全面覆盖原则 + 等同原则）
- `--无效` —— 权利要求技术特征 vs. 现有技术（新颖性/创造性）
- `--审查` —— 审查他人已产出的对照表

## 补充专利模式信息

- **专利号和主张的权利要求。** 哪些独立、哪些从属。
- **申请日/优先权日。** 确立现有技术截止日。
- **已有权利要求解释。** 已生效的解释裁定、双方确认的解释。

## 专利模式工作流程

### 第一步：解析权利要求

将主张的独立权利要求解析为编号技术特征。处理：

- **前序部分。** 注意是否构成限定——属权利要求解释问题。如解释裁定未解决，标注 `前序部分限定性: 待定`。
- **功能性限定** —— 每个"用于[功能]的[装置/模块]"或非结构性功能性术语。保护范围为说明书中公开的结构及其等同。引用对应结构（栏/行）。如说明书未公开结构，标注 `功能性限定不明确`。
- **马库什组、Jepson型权利要求、产品-by-process、方法步骤顺序依赖** —— 标注并说明特殊解释规则。
- **从属权利要求** —— 引用其父项；仅对照附加技术特征。**执行，不示意。** 如主张的权利要求包含从属项，在第四步中为每项从属项产出实际附加技术特征行——不得仅附注说明"应对从属项进行对照"。

向用户展示解析结果，确认后再映射。错误的解析会污染下面每一行。

### 第二步：权利要求解释检查

标注争议术语：

- 自创术语或说明书中定义的术语
- 有审查历史的术语（修改、陈述、放弃——禁止反悔）
- 功能性语言（"被配置为""适于""可操作以"）
- 相对术语（"基本上""约"）——明确性风险

对每项标注术语，陈述该映射在哪种解释下成立、哪种解释下不成立。

### 第三步：技术特征比对（全面覆盖原则 + 等同原则）

逐项逐目标：

1. **寻找证据。** 被控侵权产品：技术文档、产品手册、源代码、拆解报告、勘验笔录、鉴定意见。现有技术：栏/行（专利），段落（公开申请），页/图（非专利文献）。
2. **逐字引用。** 一字不差。不转述。
3. **定性映射。**

   | 映射 | 含义 | 适用场景 |
   |---|---|---|
   | `相同` | 被控侵权技术特征与权利要求技术特征完全相同 | 侵权（全面覆盖） |
   | `等同` | 基本相同的手段、功能、效果，无需创造性劳动即可联想 | 侵权（等同原则） |
   | `不相同不等同` | 技术特征不同且不构成等同 | 侵权 |
   | `缺少技术特征` | 被控侵权方案缺少权利要求中某技术特征 | 侵权（不落入保护范围） |
   | `多余指定` | 注意：中国不采多余指定原则 | 侵权 |
   | `解释依赖` | 取决于争议术语如何解释 | 均可 |
   | `字面对应` | 现有技术公开了权利要求技术特征 | 无效（新颖性） |
   | `新颖性` | 每项技术特征在单篇文献中，以权利要求所述方式排列 | 无效 |
   | `创造性-组合` | 第二篇文献补足缺失技术特征；需有组合启示 | 无效 |
   | `部分` | 部分技术特征存在 | 均可 |
   | `未找到` | 技术特征不存在 | 均可 |
   | `待证` | 从现有材料无法判断 | 均可 |

4. **每格状态。** `相同` / `等同` / `不相同不等同` / `缺少` / `部分` / `未找到` / `待证` / `解释依赖` / `新颖性` / `创造性-组合`。
5. **标注待决问题。** "如果[X]成立则本映射成立。需要[拆解/源代码/勘验/鉴定]来确认。"

**禁止静默补充。** 文档不足意味着 `待证`，而非从类似产品外推。

### 第四步：从属权利要求——执行，不示意

对每项主张的从属权利要求，产出实际行（或行组），对照目标映射附加技术特征。**产出实际行，而非占位说明"应产出行"。**

### 第四点五步：等同补充——执行，不示意

对每项被对照为 `相同` 但被控侵权特征结构相似但非完全字面对应的技术特征，产出**配对等同候选行**（侵权模式）。不得以"等同分析为单独事项"作为脚注而不产出实际等同映射。

### 第五步：全面覆盖原则判定（仅侵权）

在逐项技术特征比对完成后，作出全面覆盖原则判定：

- **全部技术特征相同** → 落入保护范围（字面侵权）
- **全部技术特征相同或等同** → 落入保护范围（等同侵权）
- **缺少一个以上技术特征** → 不落入保护范围
- **有一个以上技术特征不相同也不等同** → 不落入保护范围

标注禁止反悔和捐献规则的影响。

### 第六步：无效比对（仅无效）

- **新颖性**：每项技术特征在单篇对比文件中公开——全部技术特征在单篇文件中的，缺乏新颖性
- **创造性**：主对比文件+第二对比文件+有记录的组合启示——三步法（确定最接近现有技术→区别特征→实际解决的技术问题→是否显而易见）
- **其他无效理由**：
  - 说明书公开不充分（《专利法》第26条第3款）
  - 权利要求不清楚（《专利法》第26条第4款）
  - 修改超范围（《专利法》第33条）
  - 不属于授权客体（《专利法》第25条）

### 第七步（审查子模式）：审查

逐行：映射是否有支撑？精确引用是否准确？技术特征是否完全对口？最强反驳是什么？反驳机会在哪里？按行输出判定（`有支撑` / `薄弱` / `无支撑`）及对照表的薄弱点。

# 模式2 —— 民事要件事实对照表

## 法律方法：请求权基础分析法

将诉因（或抗辩）的法律要件与证据逐项对应。采用王泽鉴先生提出的**请求权基础分析法**：明确请求权基础规范→分解法律要件→案件事实与要件逐项比对→得出结论。

核心产出为：
1. 显示哪项证据对应哪项要件的对照表
2. 缺口清单——告诉律师缺失什么
3. 举证责任分配——谁对哪项要件承担举证责任

## 工作流程

### 第一步：识别诉请与请求权基础

- **什么诉因？** （或抗辩？）如多项诉请，逐项单独对照。
- **请求权基础是什么？** 具体到法条——如"《民法典》第577条（违约责任）"、"《民法典》第1165条（过错侵权）"、"《民法典》第235条（返还原物请求权）"。
- **哪方立场？** 原告的初步证明案件、被告的积极抗辩、被告对原告诉请的质疑。阅读 `## 立场` 获取默认值。开始前确认本案立场匹配。
- **哪个管辖区？** 省份和法院。**要件和裁判要旨语言因管辖区而异。** 模板库为基准；有约束力的裁判要旨或法条为准。
- **哪个诉状？** 加载起诉状/反诉状/答辩状，使对照表追踪实际提出的诉请，而非通用版。

### 第二步：加载要件——请求权基础的构成要件

三条路径：

**(a) 模板库。** 参考 `references/民事要件模板.md`（本技能目录下）。常见请求权基础的基准要件：

- 违约责任（《民法典》第577条）：合同有效成立、被告未履行/履行不符合约定、原告有损失、因果关系（损失因违约造成）
- 过错侵权（《民法典》第1165条）：加害行为、损害后果、因果关系、过错
- 无因管理（《民法典》第979条）：管理他人事务、无约定或法定义务、为他人利益管理
- 不当得利（《民法典》第985条）：一方获利、另一方受损、获利无法律根据、获利与受损有因果关系

**(b) 自定义。** 用户定义要件，或粘贴裁判要旨/法条/起诉状诉请进行解析。

**(c) 积极抗辩。** 同样支持抗辩对照——诉讼时效（《民法典》第188条）、不可抗力（《民法典》第180条）、受害人故意（《民法典》第1174条）、受害人过错（《民法典》第1173条）、正当防卫（《民法典》第181条）、紧急避险（《民法典》第182条）等。

**按管辖区的具体表述——主动呈现。** 如执业画像或 `案件档案.md` 列出具体管辖，主动呈现该管辖区的具体裁判要旨——不要让用户先说"你的管辖区是否添加/删除/修改"，再回复。技能应主动提供并让用户选择。

确认要件清单后再映射。如用户的管辖区不在常见范围，询问："贵管辖区的裁判要旨对这些要件是否有添加/删除/修改？"如有，使用用户版本。

### 第三步：举证责任分配

在逐项映射证据前，先明确举证责任：

- **原告**对请求权发生的要件事实承担举证责任（《民事诉讼法》第67条、《民事诉讼证据规定》第1条）
- **被告**对权利妨碍、权利消灭、权利限制的抗辩事实承担举证责任
- 法律要件分类说（罗森贝克规范说）——被中国司法实践普遍采用

| 要件 | 举证责任方 | 证明标准 |
|---|---|---|
| [要件1] | 原告 | 高度盖然性（《民诉法解释》第108条） |
| [要件2] | 原告 | 高度盖然性 |
| ... | ... | ... |

### 第四步：映射证据

逐项要件：

- **支持的证据** —— 什么证明本要件？引用来源附精确引用。
  - 书证 —— `[证据编号0001 第X页]`
  - 电子数据 —— `[证据编号0002，微信聊天记录截图 第X页]`
  - 证人证言 —— `[张三证言 庭审笔录第X页]`
  - 当事人陈述 —— `[原告陈述 庭审笔录第X页]`
  - 鉴定意见 —— `[XX司法鉴定中心[202X]鉴字第XX号 第X页]`
  - 勘验笔录 —— `[勘验笔录 第X页]`
  - 生效裁判文书 —— `[([202X]X民初X号) 第X页]`
- **逐字引用** —— 证据为书证或电子数据的，逐字引用。不转述。
- **矛盾证据** —— 什么指向相反方向？引用。这是该行的薄弱点。
- **强度** —— `强` / `中` / `弱` / `无`。简洁为上。过度校准的强度评分是噪音；`弱` 和 `无` 才是关键行。
- **每格状态** —— `有支撑` / `部分` / `有争议` / `缺口` / `待举证`。

### 第五步：缺口检测——核心产出

映射后，产出缺口清单。这就是对照表的要义。

> **证据薄弱或缺失的要件：** [列表]
>
> - 如为主张方（原告）：这些将影响诉讼请求能否得到支持。在庭审前弥合或接受败诉风险。
> - 如为抗辩方（被告）：这些是原告诉请的薄弱点——原告须证明每项要件达到高度盖然性标准；缺口即抗辩优势。
> - 如在举证阶段：这些是举证优先项——将缺口转为 `有支撑` 或确认 `无`。

缺口检测不是对实体问题的结论。是案件薄弱点的地图。

### 第六步：阶段感知框架

询问阶段。同一对照表，不同输出框架：

- **起诉前/立案。** 起诉状是否满足《民事诉讼法》第122条起诉条件？每项要件是否有初步证据支持？
- **举证阶段。** 对每项 `缺口` 或 `待举证` 要件，需要什么证据？哪些证人、哪些书证、需要申请法院调查取证还是申请鉴定？
- **庭审准备。** 对每项要件，是否存在实质性事实争议？`有支撑` 格且无矛盾证据即为强点；`有争议` 格则为庭审争议焦点。
- **庭审。** 举证顺序。哪件证据证明要件1，谁举证，证明目的是什么。对照表变成庭审举证提纲。

### 第七步（审查子模式）：审查

对对方的起诉状/代理词/证据目录或外聘律师的初稿：每项要件，其所引证据是否真正证明了该要件？何处对照表薄弱？你的最强反驳是什么？

## 共享护栏——核查清单

- **引用核实。** 每项精确引用是对来源的主张。律师核实。技能不编造引用——如果引用不能产出，该格为 `待证` 或 `缺口`。
- **来源标注。** 每条逐字引文在配套CSV中有其来源。无来源的引文不是证据。
- **禁止静默补充。** 证据薄弱意味着 `待证`/`缺口`，不是"外推"。
- **案件工作空间检查。** 写入前确认当前案件。
- **决策姿态。** 不确定要件是否满足时，标注；不决定。
- **公式注入。** 写入CSV/XLSX/表格的每个单元格检查首字符并在必要时添加前缀 `'`。默认：先中和后写入。
- **要件因管辖区而异。** 模板库为基准。有约束力的裁判要旨或法条为准。
- **对照表不是法律文书。** 每项输出为初稿。
- **全面覆盖原则的应用。** 缺少一个以上技术特征即不落入保护范围。等同须满足"三个基本相同+无需创造性劳动"标准。
- **请求权基础的识别。** 同一事实可能对应多个请求权基础（请求权竞合）。逐项单独分析。
