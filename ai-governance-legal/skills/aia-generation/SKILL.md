# 人工智能安全评估（算法安全评估）

## 前置参考资料

1. 先读取 `../../references/ai-governance-core.md`，再执行 AI 安全评估、法规对照和影响评估结论。

## 程序步骤

1. 读取 `~/.claude/plugins/config/claude-for-legal-cn/ai-governance-legal/CLAUDE.md`。确认评估模板已填充。
2. 通过以下框架确定评估等级（标准/升级/高风险）。
3. 运行信息采集——对话式，非表格式。
4. 对覆盖范围内的每个制度进行法规对照——适用义务、禁止性行为暴露和合规要求；引用法规原文。
5. 按评估模板撰写评估报告。
6. 与公司AI政策承诺进行交叉对比。
7. 产出：评估报告 + 整改条件清单 + 交接标记。

## 目的

AI安全评估是一份记录在案的合规评估，而非填空表格。它回答：这个AI系统做什么、如何产出其输出、如果出错谁受影响、有什么监督、合规状态如何。

**AI安全评估与个人信息保护影响评估不同：**
- 个人信息保护影响评估（PIA）：问的是个人数据是否被合法处理
- AI安全评估（AIA）：问的是AI系统是否被安全、负责任地设计和部署
- 两者可以并行，但不应互相替代

## 法律框架

本评估依据：

| 法规 | 核心要求 |
|------|----------|
| **《生成式人工智能服务管理暂行办法》** | 安全评估、内容安全、训练数据合规、服务透明度、用户权益保护 |
| **TC260-003《生成式人工智能服务安全基本要求》** | 语料安全、模型安全、安全措施、个人信息保护、知识产权保护的技术标准 |
| **《互联网信息服务深度合成管理规定》** | 深度合成内容标识与管理 |
| **《互联网信息服务算法推荐管理规定》** | 算法推荐服务规范、算法备案 |
| **《科技伦理审查办法（试行）》** | 科技伦理审查要求 |
| **《个人信息保护法》** | 自动化决策、个人信息处理 |

## 第零步：需要安全评估吗？

检查是否需要完整评估：

- [ ] 此AI系统是否面向中国境内公众提供服务？
- [ ] 此AI是否做出或实质影响涉及个人的决策？
- [ ] 此AI是否处理个人数据？
- [ ] 是否是面向客户的AI系统？
- [ ] 是否使用第三方AI模型且公司是服务使用者？
- [ ] 是否涉及深度合成内容（图像、音频、视频、文本生成）？
- [ ] 用例是否在高治理风险等级？

如上述皆否且不面向公众提供服务：告知无需完整评估，但提供一份简要文件记录说明原因。

## 第一步：风险等级确定

信息采集前确定风险等级和评估深度：

| 风险等级 | 条件 | 评估深度 |
|----------|------|----------|
| **标准** | 内部使用、低影响、非敏感领域 | 精简评估（核心九项） |
| **升级** | 面向客户、涉及个人信息、中等影响 | 完整评估 |
| **高风险** | 涉及敏感个人信息、自动化决策影响重大权益、涉及深度合成可造成社会影响 | 完整评估 + 持续监测方案 |

## 第二步：信息采集

以对话形式获取以下答案：

### 2.1 服务信息
- AI系统/服务的名称和用途
- 服务的提供方式（API/网页/App/嵌入式）
- 目标用户群体与规模
- 部署阶段：规划中/试点/已上线/已规模化
- 服务面向的地域（中国境内/境外/均有）

### 2.2 语料安全
- AI模型的训练数据来源（自有数据/公开数据/第三方授权/用户输入）
- 训练数据的规模与类型（文本/图像/音频/视频/代码/其他）
- 训练数据中是否包含个人信息？如是，是否获得授权或告知同意？
- 训练数据中是否包含受著作权保护的内容？是否有合法授权？
- 是否有训练数据的过滤和审核机制（过滤违法和不良信息）？
- 是否有训练数据来源的文档记录和安全性评估？

### 2.3 模型安全
- 使用什么模型（自研/第三方基础模型/开源模型）？
- 模型提供方和版本
- 模型是否经过安全性测试？测试结果如何？
- 是否有对抗性攻击的防护措施？
- 模型是否支持生成违法和不良信息？是否有内容过滤机制？
- 生成内容的准确性如何评估？

### 2.4 安全措施
- 输入内容审核（用户输入是否经过安全过滤）
- 输出内容过滤（AI生成内容是否经过安全过滤）
- 内容安全的关键词库和模型是否持续更新
- 是否有用户投诉举报机制（《生成式AI服务管理暂行办法》第12条）
- 是否有应急响应预案（发现违法内容时的处置流程）
- 是否对深度合成内容依法进行标识

### 2.5 服务透明度
- 是否在服务协议中明确说明AI服务的性质和使用限制？
- 是否以显著方式提示用户其正在与AI系统交互？
- 如涉及深度合成，是否对生成内容进行标识？
- 是否向用户提供拒绝自动化决策的选项？

### 2.6 个人信息保护
- AI系统是否处理个人信息？如是，处理的目的和范围
- 是否在隐私政策中说明AI处理个人信息的情况？
- 如涉及自动化决策，是否提供说明和拒绝权？
- 是否涉及敏感个人信息处理（单独同意的获取情况）

### 2.7 科技伦理
- 是否进行了科技伦理审查（《科技伦理审查办法（试行）》）？
- 是否建立了科技伦理审查委员会或有指定负责人？
- AI系统是否在以下高风险伦理领域：
  - 涉及人类生命健康和安全
  - 涉及大规模个人信息和隐私
  - 涉及社会公平正义（如教育/就业/信贷）
  - 涉及弱势群体权益
  - 涉及人类自主性和尊严

### 2.8 知识产权
- AI生成内容的知识产权归属
- 训练数据中是否涉及第三方知识产权
- 是否建立了知识产权侵权投诉处理机制
- 用户输入内容的知识产权归属约定

**评估结论级别：**
- **符合：** 完全满足合规要求，无需整改
- **基本符合：** 主体满足但存在个别需要完善之处，列出整改项
- **不符合：** 存在重大合规差距，需整改后重新评估

## 风险质量标准

风险必须**具体且与系统设计挂钩**。目标2-5个真实风险，而非12个填充项。每个风险必须：
- 描述具体的场景（不是"模型可能出错"，而是"在何种输入下模型会产生何种错误输出"）
- 链接到具体的法规义务
- 有对应的缓解措施（不能只发现问题不提解决办法）

## AI政策对比

每项评估应与公司AI政策承诺交叉检查。标记每个不匹配项。如公司政策比法规要求更严格，以公司政策为准。如法规要求更严格，以法规为准。

## 本技能不做的事

- 不批准部署。由有权限的人签署评估报告。
- 不构成法定安全评估的全部内容——面向公众的生成式AI服务可能需要向主管部门另行报送。
- 不设计具体的缓解措施技术方案——仅从合规角度提出要求，由工程团队设计实现方案。
- 不替代涉及个人信息处理时的个人信息保护影响评估（PIA）。
