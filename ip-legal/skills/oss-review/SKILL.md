# /oss-review

对照执业画像中的开源政策，在**中国法律环境**下运行开源许可证合规检查。按许可证族分类依赖项、将义务映射到部署模型（SaaS/本地部署/分发）、标记许可证未知和非OSI的源码可用（source-available）包、识别木兰系列许可证的中国自主开源许可证兼容性问题、并建议行动——遵守、替换、移除、寻求法律审查、寻求商业许可。

## 指示

1. 加载执业画像和任何已上传的开源政策。
2. 确定审查范围（全量依赖列表/单个组件/准备发布的代码仓库）。
3. 在分类义务前确定部署模型（SaaS 网络服务/本地部署分发/嵌入式设备）。
4. 遵循工作流：读取实际许可证文本而非仅元数据、按许可证族分类每个包、映射义务到部署模型、标记失败模式。
5. 输出审查备忘录。

## 中国法环境下的开源合规框架

### 开源许可证法律效力
在中国司法实践中，开源许可证的法律效力逐步得到确认。许可证通常被分析为：
- **格式合同**（依据《民法典》第496-498条格式条款规则）——许可证文本为权利人预先拟定，未经逐个协商
- 许可证条款不得排除或限制被许可人的主要权利（格式条款无效规则）
- 对格式条款有两种以上解释的，应作出不利于提供方的解释

### 关键考量
- **GPL传染性（copyleft）**：在中国司法实践中逐步被认可。软件分发场景下的copyleft义务，包括衍生作品的源代码披露义务，可能被法院认可为合同义务。AGPL在网络使用（SaaS）场景下的义务延伸需特别关注。
- **《民法典》合同编**：许可证条款的订立、履行、违约责任受合同法规则调整
- **网络安全审查**：关键信息基础设施运营者使用的开源组件可能涉及网络安全审查（《网络安全审查办法》）
- **《网络安全法》**：开源组件中的安全漏洞可能触发网络安全义务
- **出口管制**：涉及加密算法的开源软件可能受《密码法》和商用密码管理规定约束
- **数据安全**：开源软件中所嵌入的数据收集功能可能触发《个人信息保护法》和《数据安全法》义务

### 中国信息通信研究院（信通院）标准
- **《开源许可证兼容性指南》**：信通院发布的开源许可证之间的兼容性参考指南
- **《开源软件治理能力评价方法》**：对组织的开源治理成熟度进行评价
- **《开源元数据》：** 开源项目的基础信息描述标准
- **中国开源生态**：木兰系列（MulanPSL-2.0、MulanPubL-2.0等）为中国自主知识产权的开源许可证，在信通院指导下设计，具有法律文本中文化、符合中国司法实践的特点

## 审查流程

### 步骤1：确定范围
- 全量依赖清单（package.json / requirements.txt / pom.xml / go.mod等）
- 单个组件的许可证审查
- 输出代码开源前的合规检查（代码中是否含他人代码、第三方库、复制粘贴的代码段）

### 步骤2：确定部署模型
部署模型决定copyleft义务的触发范围：

| 部署模型 | GPL传染风险评估 | AGPL传染风险评估 |
|---|---|---|
| **本地部署分发**（用户安装运行） | 需披露衍生作品源码 | 同GPL |
| **SaaS 网络服务**（用户通过网络访问） | 通常不需披露源码（ASP loophole） | **需披露源码**（AGPL关闭了ASP漏洞） |
| **嵌入式设备** | 需披露源码 + 安装说明 | 同GPL且更严格 |

**【注意】** 中国司法实践尚未就SaaS场景下GPL/AGPL的传染义务范围形成明确判例，但国际趋势和学术观点倾向于对copyleft许可证按文本解释。

### 步骤3：读取实际许可证文本
不依赖包管理器的许可证元数据（可能不准确）。从源码仓库或许可证文件直接读取完整许可证文本。特别注意：
- 许可证是否有例外条款（如GPL+linking exception、GPL+Classpath exception）
- 是否包含FLOSS Exception（允许与特定开源许可证链接）
- 是否声明"or later version"（GPL-3.0-only vs GPL-3.0-or-later）

### 步骤4：按许可证族分类

| 桶 | 许可证 | 关键义务 |
|---|---|---|
| **宽松型** | MIT、BSD（2-Clause/3-Clause）、Apache-2.0 | 保留著作权声明和许可证文本 |
| **弱Copyleft** | LGPL-2.1/3.0、MPL-2.0、EPL-2.0 | 文件/库层级源代码披露；LGPL允许专有软件链接 |
| **强Copyleft** | GPL-2.0、GPL-3.0 | 衍生作品的完整源代码披露义务 |
| **网络强Copyleft** | AGPL-3.0 | 网络使用时也触发源代码披露义务 |
| **公共领域/放弃** | CC0、Unlicense、WTFPL | 通常无义务（Unlicense在中国法下的弃权效力需评估） |
| **非OSI源码可用** | SSPL、BUSL、Elastic License、Redis Source Available License | 非开源——限制商业使用/竞争性使用 |
| **中国开源许可证** | MulanPSL-2.0、MulanPubL-2.0 | 中文文本，符合中国法律语境；MulanPSL宽松型，MulanPubL弱Copyleft |
| **其他/自定义/未知** | 供应商特定、自拟条款、缺失许可证 | **停止——不要默认视为宽松** |

### 步骤5：映射义务到部署模型

逐包走查：该许可证义务在当前的部署模型下是否触发？

| 许可证 | 本地分发 | SaaS | 嵌入式 |
|---|---|---|---|
| MIT/BSD/Apache-2.0 | 保留声明 | 保留声明 | 保留声明 |
| LGPL | 库源码披露 | 同左 | 库源码+链接说明 |
| GPL-2.0/3.0 | 衍生作品源码披露 | 通常不触发 | 完整源码+安装说明 |
| AGPL-3.0 | 衍生作品源码披露 | **需源码披露** | 完整源码+安装说明 |
| MulanPSL-2.0 | 保留声明 | 保留声明 | 保留声明 |
| MulanPubL-2.0 | 修改文件源码披露 | 同左 | 修改文件源码披露 |

### 步骤6：标记失败模式
- 许可证不兼容：两个依赖的许可证之间存在冲突（如GPL-2.0和Apache-2.0）
- copyleft传染：强copyleft许可证在当前的部署模型下触发源码披露义务
- 许可证缺失/未知：无LICENSE文件的包
- 非OSI源码可用：SSPL/BUSL/Elastic License等限制竞争性使用的许可证
- 许可证冲突：声明为MIT但代码中包含GPL组件
- 木兰系列和其他许可证的兼容性（信通院《开源许可证兼容性指南》）

### 步骤7：输出代码开源前检查（如适用）
如果准备将自有代码以开源许可证发布：
- 检查是否含有他人的专有代码
- 检查是否含有他人开源代码（标注来源和许可证）
- 确认所选许可证与嵌入的第三方代码的许可证兼容

## 决策立场

当许可证不能自信分类时，标记为"需要审查"——不要称为宽松。低报许可证风险是一扇单向门。

## 中国许可证特别说明

**木兰系列许可证（Mulan Series）：**
- **MulanPSL-2.0（宽松型）**：类似Apache-2.0的中文宽松许可证，明确专利授权条款，符合中国法律表达习惯
- **MulanPubL-2.0（弱Copyleft）**：要求将修改后的源代码以木兰许可证或兼容许可证发布
- 与其他许可证的兼容性见信通院《开源许可证兼容性指南》

木兰许可证与GPL系列许可证的兼容性问题需特别关注——信通院指南中对此有分析。

## 此技能不做什么

- 不提供正式开源合规法律意见。审查备忘录为内部参考。
- 不决定某一许可证的最终法律效力（在中国司法实践中仍在发展）。
- 不撰写开源政策。遵从执业画像中的内部政策。
- 不评估开源许可证之外的IP风险（如专利侵权）。
